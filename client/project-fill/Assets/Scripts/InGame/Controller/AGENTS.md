## Rules
- Controller owns game logic; BoardView owns rendering/animation. Controller never touches GameObjects.
- Booster economy: Undo free, single-step (no inventory, no count label; only the last move is reversible — `Board.CanUndo` greys the button after one undo or with no prior move); Shuffle + AddLane consume OWNED inventory items via `ConsumeThen` (server-authoritative `/api/inventory/spend`); the booster-bar Label shows the held quantity (`× N` from `PlayerProgressService.GetItemCount`) when owned, else the **gold price** (`item.csv`) to signal instant-buy. When owned count is 0, tapping opens a **confirm popup** (`BuyBoosterThenUse`); on confirm it buys one with gold (`/api/inventory/buy`, server-authoritative) and uses it; insufficient gold → `toast.insufficient_gold`. Item ids + prices from `BoosterType.ItemId()` / `ItemDataService`, never hardcoded. The Hard-Stuck fail overlay has **no Shuffle** — its only rescue is the free ad-reward AddLane (§5.1/§5.4); otherwise Retry (restart) / Forfeit (lobby).
- Stage clear is server-authoritative: `SubmitClear` POSTs `StageApiService.ClearStage` (moves + completed signal types `{0..types-1}`); server owns best-moves, ranking, first-clear / chapter-chest rewards, and achievement progress. Response syncs gold (in API service) + caches best-moves and unlock (`ApplyMaxClearedStage`). Local fallback (RecordBestMoves + UnlockStage) when the API is absent. Star rating removed.
- Ads (spec §5.4): rewarded `STUCK_ADD_LANE` gates the stuck Add Lane; `_adRewardedThisStage` suppresses the post-stage interstitial; flag resets per stage in `LoadCurrent`. All ad calls null-guard the service (dev/no-Boot play still works).
- BoardView events are subscribed once in `Begin` (BoardView persists across stage reloads).
- Campaign entry is server-gated: `InGameSceneEntry` calls `StageApiService.StartStage` and only calls `_controller.Begin(index, SessionId)` on success; STAGE_LOCKED bounces back to Lobby. **Every stage attempt goes through start** — the result-screen "Next" (`AdvanceToNextStage`) re-runs `StartStage` for a fresh `SessionId` before building the next board, NOT a bare `LoadCurrent`. The clear echoes that `_sessionId`; a server `INVALID_STAGE_ATTEMPT` rejection forces the player to the lobby via `ShowSessionEndedAndExit`. Offline/dev (no `StageApiService`) bypasses the gate and `Begin`/advance locally (no session).
- Move flow: capture top chip + `MovableCount` + destBase → `Board.Move` (A-R08 stack pour) → `BoardView.AnimateMove(…, onComplete: PostMoveCheck + DrainTapQueue)`.
- Tap buffering: `Board.Move` resolves move+absorb atomically while `AnimateMove` replays it visually with input blocked (`_inputBlocked`). To keep speed without model/view desync, lane taps during a **move** animation are queued (B1: tap-event buffer, view stays truthful, cap = 1 lookahead move) and replayed in order on completion. `BoardView` lane-tap event always forwards; the controller decides enqueue (only when `_animating`) vs process. Modal blocks (ad/network spend) drop taps as before; boosters are never queued.
- Tutorial: `LoadCurrent` calls `TutorialManager.NotifyBoardReady(stageId, DetectGimmicks(_def))` after board build (FirstLaunch + GimmickAppear); hard-stuck increments `_failCount` → `NotifyFailRepeat`. `HandleLaneTapped` gates on `BoardInputBlocked` (blocks only Tap/info steps; Select/Move steps play for real) and fires `NotifyAction(Select)` on selection + `NotifyAction(Move)` on move-complete so interactive steps advance. `HandleBooster`/`HandlePause` gate on `ActiveBlocking` (always blocked while a tutorial runs). Pause "How to play" → `TutorialManager.Recap`.
- `LoadCurrent` → `ResolveRow(index)` reads `StageDataService.GetStage(index+1)`; `BuildDefinition` maps gimmicks + `difficulty` to `StageDefinition` and names it `STAGE {stage_id}` (unified global number, not chapter-order); the board comes from `BoardCodec.Decode(row.board, def)`. Empty `board`/no row → `BoardFactory.Generate` fallback; null → `StageLibrary`. Glyph order `RBGYPCOMLT`=SignalType 0..9; lane codes N/L/B.
